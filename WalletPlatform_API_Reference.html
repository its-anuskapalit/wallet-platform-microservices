Overview & Architecture

System design, service map, and routing conventions

WalletPlatform is a production-grade, .NET 10 microservices fintech system. All client requests are routed through an Ocelot API Gateway on port 5000, which handles JWT validation and downstream routing. Each service owns its dedicated SQL Server Express database, communicates asynchronously via RabbitMQ, and delivers notifications through MailKit.

Service Registry

API Gateway (Ocelot)

:5000

Single entry point. JWT validation, rate limiting, request routing to all downstream services.

Auth Service

:5001

User registration, login, JWT issuance, KYC submission. Publishes UserRegistered & KYCStatusUpdated events.

User Profile Service

:5002

User profile retrieval and management. Reads user data from Auth DB context or its own store.

Wallet Service

:5003

Wallet CRUD, balance management, P2P transfers, top-up, transaction history. Publishes TransactionCompleted & TransactionFailed.

Ledger Service

:5004

Immutable double-entry accounting ledger. Subscribes to transaction events, records debits/credits.

Rewards Service

:5005

Points accrual, tier management (Bronze/Silver/Gold), reward history. Subscribes to TransactionCompleted.

Catalog Service

:5006

Redeemable offers, deals, and catalog items. Manages product listings available for reward redemption.

Notification Service

:5007

Multi-channel notifications (in-app + MailKit/Gmail SMTP). Subscribes to all platform events.

Receipts Service

:5008

Generates and stores transaction receipts. Subscribes to TransactionCompleted events.

Admin Service

:5009

KYC review & approval, user management, wallet freeze/unfreeze. Role-gated to Admin JWT claims.

Authentication

JWT Bearer token scheme used across all protected endpoints

WalletPlatform uses JWT (JSON Web Token) Bearer authentication. Upon successful registration or login, you receive a signed token that must be included in the Authorization header of every protected request. The Ocelot Gateway validates the token signature before forwarding.

Request Header

Authorization: Bearer <your_jwt_token>

JWT Token Claims

ClaimTypeDescription
nameidentifierGuidUnique user ID across all services
FullNamestringUser's full display name
EmailstringUser's registered email address
RoleUser | AdminAuthorization role — controls access to admin endpoints
statusPending | Active | RejectedKYC approval status — Active required for wallet ops
expUnix timestampToken expiry — 8 hours from time of issue

Standard Response Wrapper

Every endpoint returns a consistent JSON envelope regardless of success or failure:

Success
// HTTP 200 OK
{
  "success": true,
  "message": "Operation successful.",
  "data": { /* response payload */ }
}
Failure
// HTTP 4xx / 5xx
{
  "success": false,
  "message": "Descriptive error message here.",
  "data": null
}
ℹ️ Token is valid for 8 hours. Re-authenticate via POST /auth/login when expired (HTTP 401 returned).

Async event-driven communication between services

WalletPlatform uses RabbitMQ for asynchronous inter-service communication. Services publish events to named exchanges; downstream services subscribe and react independently.

Platform Events

Event NameExchangePublished ByConsumed ByTrigger
